# Insurance Integration Hub

보험사 금융 IT 환경에서 외부 기관과의 인터페이스 연계 상태를 관리하고  
장애 발생 시 운영자가 빠르게 대응할 수 있도록 설계한 관리자 대시보드 MVP입니다.

실제 서비스 전체 구현이 아닌,  
운영 시스템에서 필요한 핵심 기능을 중심으로 MVP 형태로 개발했습니다.

---

## 프로젝트 개요

보험사 내부 시스템은 카드사, 은행, 공공기관 등 다양한 외부 기관과 데이터를 주고받습니다.

이 과정에서 인터페이스 실패, 처리 지연, 로그 누락 등의 문제가 발생하면  
고객 서비스 장애나 운영 리스크로 이어질 수 있습니다.

운영자가 전체 연계 상태를 한눈에 확인하고,  
실패 건을 빠르게 파악하며, 실행 이력을 추적할 수 있도록 관리자 시스템을 구현했습니다.

---

## 주요 기능

### 운영 대시보드

- 전체 인터페이스 상태를 한눈에 확인할 수 있도록 구성
- 성공 / 실패 / 대기 상태 시각화
- 장애 발생 건 우선 확인

### 외부 기관 관리

- 외부 기관 등록 / 조회 / 수정 / 삭제
- 기관명 중복 등록 방지

### 인터페이스 관리

- 인터페이스 등록 / 조회 / 수정
- 외부 기관과 인터페이스 연결 관리
- 인터페이스명 중복 체크

### 실행 로그 관리

- 실행 이력 조회
- 상세 로그 확인
- 페이지네이션 적용

### 운영 자동화

- 장시간 `PENDING` 상태 자동 점검 스케줄러

## 상태 정책

인터페이스 상태는 사용자가 직접 입력하는 값이 아니라  
실행 흐름과 결과를 기준으로 자동 관리됩니다.

| 상태 | 의미 |
|------|------|
| PENDING | 실행 대기 중인 상태 |
| RUNNING | 현재 실행 중인 상태 |
| SUCCESS | 정상 실행 완료 |
| FAILED | 실행 실패 또는 장애 발생 |

### 상태 변경 흐름

- 실행 요청 접수 시 → `PENDING`
- 실제 처리 시작 시 → `RUNNING`
- 정상 완료 시 → `SUCCESS`
- 실행 중 오류 발생 시 → `FAILED`

### 운영 관점 활용

- `FAILED` 상태는 장애 대응 대상 건으로 우선 확인합니다.
- `PENDING` 상태가 장시간 유지되면 지연 가능성으로 판단합니다.
- `RUNNING` 상태를 통해 현재 처리 중인 작업을 확인할 수 있습니다.
- 운영자는 대시보드에서 전체 상태를 한눈에 확인할 수 있도록 구성했습니다.

## 주요 구현 포인트

### 1. 운영 관점 상태 관리

단순 CRUD가 아니라 운영 시스템 특성을 반영하여  
인터페이스 상태를 `PENDING`, `RUNNING`, `SUCCESS`, `FAILED`로 분리했습니다.

이를 통해 운영자는 실행 대기, 실행 중, 정상 완료, 실패 상태를 구분하여 확인할 수 있습니다.

### 2. 실행 로그 이력 관리

운영자가 장애 원인을 추적할 수 있도록  
실행 로그 조회 및 상세 확인 기능을 구현했습니다.

각 실행 이력에는 실행 상태, 실행 시간, 에러 메시지 등을 포함하여  
실패 발생 시 원인 파악이 가능하도록 구성했습니다.

### 3. 페이지네이션 적용

실행 로그는 지속적으로 누적될 수 있는 데이터이므로  
전체 조회 방식이 아닌 페이지네이션 기반 조회를 적용했습니다.

백엔드 API와 프론트엔드 UI 모두 페이지 단위 조회가 가능하도록 연동했습니다.

### 4. 중복 데이터 방지

외부 기관명과 인터페이스명이 중복 등록되지 않도록  
서비스 계층에서 검증 로직을 구현했습니다.

중복 등록 시 예외를 발생시켜 잘못된 운영 데이터가 쌓이지 않도록 했습니다.

### 5. 장기 대기 상태 자동 점검

`PENDING` 상태가 장시간 유지되는 경우를 운영 리스크로 보고,  
이를 자동으로 점검하는 스케줄러를 구현했습니다.

이를 통해 운영자가 수동으로 모든 대기 상태를 확인하지 않아도  
지연 가능성이 있는 인터페이스를 관리할 수 있도록 했습니다.

## API 문서

Springdoc OpenAPI를 활용하여 Swagger 기반 API 문서를 제공했습니다.

이를 통해 프론트엔드와 백엔드 간 API 명세를 쉽게 확인하고,  
요청/응답 구조를 테스트할 수 있도록 구성했습니다.

## 주요 API

### 외부 기관

| 기능 | Method | URL |
|------|--------|-----|
| 외부 기관 등록 | POST | `/api/organizations` |
| 외부 기관 조회 | GET | `/api/organizations` |
| 외부 기관 수정 | PUT | `/api/organizations/{id}` |
| 외부 기관 삭제 | DELETE | `/api/organizations/{id}` |

### 인터페이스

| 기능 | Method | URL |
|------|--------|-----|
| 인터페이스 등록 | POST | `/api/interfaces` |
| 인터페이스 조회 | GET | `/api/interfaces` |
| 인터페이스 수정 | PUT | `/api/interfaces/{id}` |
| 인터페이스 실행 | POST | `/api/interfaces/{id}/execute` |

### 실행 로그

| 기능 | Method | URL |
|------|--------|-----|
| 실행 로그 조회 | GET | `/api/execution-logs?page=0&size=10` |
| 인터페이스별 실행 로그 조회 | GET | `/api/interfaces/{id}/execution-logs` |

## 프로젝트 회고

이번 프로젝트에서는 단순 데이터 관리 화면이 아니라  
운영자가 장애 상황에서 빠르게 상태를 파악하고 대응할 수 있는 구조를 목표로 했습니다.

특히 상태 정책, 실행 로그, 페이지네이션, 장기 대기 상태 점검을 구현하면서  
운영 대시보드에서 중요한 요소가 단순 조회 기능이 아니라  
장애 대응 흐름과 운영 안정성이라는 점을 고민할 수 있었습니다.
